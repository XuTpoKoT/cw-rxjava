# RxJava
## Архитектура реализованной системы

Система реализует упрощённую модель реактивного программирования по аналогии с RxJava. Основные компоненты архитектуры:

- **`Observable<T>`** — источник данных, который может эмитить последовательность элементов типа `T`, а также сигнал завершения или ошибку. Это центральный класс, предоставляющий операторы (`map`, `filter`, `flatMap`, `subscribeOn`, `observeOn`) и метод подписки `subscribe`.

- **`Observer<T>`** — интерфейс наблюдателя с тремя методами: `onNext(T item)`, `onError(Throwable t)`, `onComplete()`. Он определяет контракт для обработки элементов потока.

- **`Disposable`** — интерфейс для управления подпиской. Позволяет явно отменить подписку через метод `dispose()`, предотвращая дальнейшую обработку элементов.

- **`Emitter<T>` / `InnerEmitter<T>`** — внутренние интерфейсы для передачи данных от источника к наблюдателю. `Emitter<T>` — фабрика, которая при подписке создаёт экземпляр `InnerEmitter<T>`, через который источник «толкает» данные.

- **`Scheduler`** — абстракция над выполнением задач в определённом потоке или пуле потоков. Определяет, где будет выполняться логика эмиссии и обработки элементов.

- **Конкретные реализации планировщиков**: `IOThreadScheduler`, `ComputationScheduler`, `SingleThreadScheduler` — используют `ExecutorService` для запуска задач в соответствующих пулах потоков.


### Структурные связи

- `Observable` инкапсулирует `Emitter`, который при подписке создаёт `InnerEmitter`.

- `InnerEmitter` оборачивается в `EmitterWrapper`, который:

    - Перенаправляет вызовы `onNext/onError/onComplete` в нужный `Scheduler` (для `observeOn`).

    - Проверяет флаг `disposed`, чтобы не обрабатывать данные после отмены подписки.

- Операторы (`map`, `filter`, `flatMap`) создают новые `Observable`, подписываясь на предыдущий и трансформируя/фильтруя элементы.

- `subscribeOn` влияет на то, в каком потоке будет вызван `emitter.emit(...)`, т. е. где источник начнёт эмитить данные.

- `observeOn` влияет на поток, в котором будут вызваны методы `Observer` (`onNext`, `onError`, `onComplete`).


### Ключевые принципы

- **Ленивость**: `Observable` не начинает эмитить данные, пока не будет вызвана подписка (`subscribe`).

- **Композиция**: операторы возвращают новые экземпляры `Observable`, позволяя строить цепочки преобразований.

- **Изоляция потоков**: `subscribeOn` и `observeOn` позволяют явно контролировать, в каких потоках выполняется эмиссия и обработка.

- **Отмена подписки**: `Disposable` даёт возможность досрочно остановить обработку потока, что важно для экономии ресурсов и предотвращения утечек.

## Принципы работы Schedulers, их различия и области применения

`Scheduler` — это функциональный интерфейс с единственным методом `execute(Runnable task)`, который отвечает за выполнение задачи в определённом контексте выполнения (потоке/пуле).

### Реализация планировщиков

1. **`IOThreadScheduler`** — использует `Executors.newCachedThreadPool()`:

    - Подходит для операций ввода‑вывода (сеть, файлы, БД).

    - Пул потоков кэширует потоки и создаёт новые по мере необходимости, что эффективно при большом числе коротких I/O‑задач.

    - В проекте применяется для `subscribeOn`, чтобы эмиссия данных происходила в отдельном потоке, не блокируя основной поток.

2. **`ComputationScheduler`** — использует фиксированный пул потоков: `newFixedThreadPool(Runtime.getRuntime().availableProcessors())`:

    - Оптимизирован для CPU‑интенсивных задач (вычисления, маппинг, фильтрация).

    - Количество потоков соответствует числу доступных процессоров, чтобы избежать избыточного переключения контекста.

    - Может использоваться для тяжёлых трансформаций в операторах.

3. **`SingleThreadScheduler`** — `newSingleThreadExecutor()`:

    - Все задачи выполняются строго последовательно в одном потоке.

    - Полезен для сценариев, где важен порядок выполнения и отсутствие параллелизма (например, обновление UI, логирование, работа с состоянием).

    - В тестах используется для `observeOn`, чтобы убедиться, что обработка элементов происходит в одном потоке.


### Различия и выбор планировщика

|Планировщик|Тип пула|Назначение|Когда использовать|
|---|---|---|---|
|`IOThreadScheduler`|Кэширующий (`newCachedThreadPool`)|I/O‑операции|Сетевые запросы, чтение файлов, обращения к БД|
|`ComputationScheduler`|Фиксированный (`newFixedThreadPool`)|CPU‑нагруженные задачи|Вычисления, сложные преобразования данных|
|`SingleThreadScheduler`|Один поток (`newSingleThreadExecutor`)|Последовательное выполнение|UI‑обновление, логирование, синхронизация состояния|

### Взаимодействие с `Observable`

- **`subscribeOn(Scheduler scheduler)`**: определяет поток, в котором будет вызван `emitter.emit(...)`. Действует один раз для всей цепочки, начиная с источника.

- **`observeOn(Scheduler scheduler)`**: меняет поток для всех последующих вызовов `Observer` (начиная с места вставки оператора). Можно использовать несколько раз в цепочке для переключения между потоками на разных этапах обработки.


## Процесс тестирования и основные сценарии

Тестирование проводилось с использованием JUnit 5. Для синхронизации асинхронных операций применялся `CountDownLatch`, а для проверки порядка и значений — коллекции `List` и утверждения `assertEquals`, `assertTrue`, `assertFalse`.

### Основные сценарии тестирования

1. **Базовая подписка (`testBasicSubscription`)**:

    - Создаётся `Observable`, который эмитит три целых числа и завершается.

    - Проверяется, что `Observer` получает все элементы в правильном порядке, а затем вызывается `onComplete`.

    - Убеждаемся, что не возникает ошибок и поток завершается корректно.

2. **Операторы `map` и `filter` (`testMapAndFilter`)**:

    - Генерируется диапазон чисел от 1 до 10.

    - Применяется `filter` для отбора чётных чисел.

    - Затем `map` удваивает значения.

    - Проверяем итоговую последовательность: `[4, 8, 12, 16, 20]`.

3. **Оператор `flatMap` (`testFlatMap`)**:

    - Для каждого числа из диапазона создаётся внутренний `Observable`, эмитящий две строки.

    - `flatMap` «расплющивает» вложенные потоки в один плоский поток.

    - Ожидаемый результат: `["A1", "B1", "A2", "B2", "A3", "B3"]`.

4. **Обработка ошибок (`testErrorPropagation`)**:

    - Источник сразу эмитит ошибку через `onError`.

    - Проверяем, что `onNext` не вызывается, `onComplete` не срабатывает, а `onError` получает ожидаемое исключение с правильным сообщением.

5. **Работа с планировщиками (`testSubscribeOnObserveOn`)**:

    - Эмиссия происходит в потоке `IOThreadScheduler`.

    - Обработка элементов переключается на `SingleThreadScheduler` через `observeOn`.

    - Фиксируются имена потоков для `emit`, `map`, `onNext`, `complete`.

    - Проверяется, что потоки действительно отличаются, подтверждая корректную работу переключения контекста.

6. **Отмена подписки (`testDisposable`)**:

    - Подписчик отменяется через `dispose()` до того, как фоновый поток попытается эмитить элемент.

    - Проверяем, что `onNext` не был вызван после отмены, демонстрируя корректную обработку флага `disposed`.


### Особенности тестового окружения

- Все тесты используют явное указание типов (`Observable.<Integer>create(...)`) для избежания проблем с выводом дженериков.

- Применяются анонимные классы `Observer<T>` для явной реализации всех трёх методов, что исключает ошибки несоответствия функциональных интерфейсов.

- Для корректной работы планировщиков в тестах вызываются методы `shutdown()` у экземпляров `IOThreadScheduler` и `SingleThreadScheduler`, чтобы освободить ресурсы пулов потоков.
